# TDD의 개념

### **❓** TDD란?

 Test Driven Development의 약자로 **'테스트 주도 개발'** 이라고 한다. 반복 테스트를 이용한 소프트웨어 방법론으로, 작은 단위의 테스트 케이스를 작성하고 이를 통과하는 코드를 추가하는 단계를 반복하여 구현한다.

짧은 개발 주기의 반복에 의존하는 개발 프로세스로 애자일 방법론 중 하나인 eXtream Programming(XP)의 ‘Test-First’ 개념에 기반을 둔 단순한 설계를 중요시한다.

```
eXtream Programming(XP) ?

미래에 대한 예측을 최대한 하지 않고,
지속적으로 프로토타입을 완성하는 애자일 방법론 중 하나이다.
이 방법론은 추가 요구사항이 생기더라도, 실시간으로 반영할 수 있다.
```

### **❗** TDD 개발주기

**Red: 실패하는 테스트 코드를 먼저 작성한다.**

**Green: 테스트 코드를 성공시키기 위한 실제 코드를 작성한다.**

**Yellow: 중복 코드 제거, 일반화 등의 리팩토링을 수행한다.** <br>

실패하는 테스트 코드를 삭성할 때까지 실제 코드를 작성하지 않고 실패하는 테스트를 통과할 정도의 최소 실제 코드를 작성해야 한다. 이를 통해, 실제 코드에 대해 기대되는 바를 명확하게 정의해 불필요한 설계를 피할 수 있다.

🔸 **일반 개발 방식**

위의 그림처럼 보통의 개발 방식은 **‘요구사항 분석 → 설계 → 개발 → 테스트 → 배포’** 형태의 개발 주기를 갖는다.

하지만 이러한 방식은 개발을 느리게 하는 위험이 존재한다.

1. 유저의 요구사항이 명확하지 않을 수 있다.
2. 처음부터 완벽한 설계는 어렵다.
3. 소스코드의 품질이 저하될 수 있다.
4. 자체 테스트 비용이 증가할 수 있다.

 
 어느 프로젝트나 초기 설계가 완벽할 수 없다. 많은 외부 또는 내부 이슈로 인해 재설계하여 점진적으로 완벽한 설계로 나아간다. 이러한 재설계로 인해 코드를 수정 하는 과정에서 불필요한 코드가 남거나 중복처리 될 수 있다.

이러한 코드들은 재사용이 어렵고 유지보수를 어렵게 만든다.

🔸 **TDD 개발 방식**

TDD와 일반적인 개발 방식의 차이점은 테스트 코드를 작성한 뒤에 실제 코드를 작성한다는 점이다. 설계 단계에서 목적과 무엇을 테스트 할지(테스트 케이스 작성) 미리 정의해야 한다.

테스트 코드 작성 도중 발생하는 이슈들은 테스트 케이스에 추가하고 설계를 개선한다. 이후 테스트가 통과된 코드만을 실제 코드로 작성한다.

이러한 단계가 반복적으로 진행되면서 자연스럽게 이슈가 줄어들고, 소스코드는 간결해진다. 또한 재설계 시간이 절감된다.
<br>
  <br>
  
### 🍀  TDD 개발 방식의 장점

- **보다 튼튼한 객체 지향적 코드 생산** <br> TDD를 통한 소프트웨어 개발 시 기능 별 철저한 모듈화가 이뤄진다. 이는 종속성과 의존성이 낮은 모듈로 조합된 소프트웨어 개발을 가능하게 하며 필요에 따라 모듈을 추가하거나 제거해도 소프트웨어 전체 구조에 영향을 미치지 않게 된다.
- **재설계 시간의 단축** <br> 테스트 코드를 먼저 작성하기 때문에 요구사항을 분명히 정의하고 개발을 시작하게 된다. 또한 테스트 시나리오를 작성하면서 다양한 예외사항에 대해 생각해볼 수 있다. 이는 개발 진행 중 소프트웨어의 전반적인 설계가 변경되는 일을 방지할 수 있다.
- **디버깅 시간의 단축** <br> TDD의 경우 자동화 된 유닛테스팅을 전재하므로 특정 버그를 손 쉽게 찾아낼 수 있다. 
- **테스트 문서의 대체 가능** <br> 테스팅을 자동화 시킴과 동시에 보다 정확한 테스트 근거를 산출할 수 있다.
- **추가 구현의 용이함** <br> 테스트 기간을 획기적으로 단축시킬 수 있기 때문에 어떤 기능을 추가하더라도 쉽게 테스트와 이슈 발견, 수정이 가능하다.
<br>

### 🍀  TDD 개발 방식의 단점

- **생산성 저하** <br> 처음부터 2개의 코드를 짜야하고, 테스트를 하면서 고쳐나가야 하기 때문에 TDD 방식의 개발 시간은 일반적인 개발 방식에 비해 대략 10~30% 정도로 늘어난다.
